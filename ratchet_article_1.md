# Ratchet: Verbs Instead of Roles

## The loop most coding agents still run on

Most production coding agents run on the same basic cycle: reason about what to do, take an action, observe the result, repeat. This pattern — usually called ReAct — is the foundation underneath LangChain's AgentExecutor and most agentic coding tools in current use. A single model holds the plan, the execution, and the judgment of whether the execution worked, all in one continuous context.

The cost structure of this pattern is well documented at this point. Retry loops compound fast: when an agent hits a test failure, tries a fix, and fails again, each retry is a full round-trip at whatever context size the session has already accumulated — not a fresh, cheap attempt. Three failed tries late in a session don't cost three times a turn; they cost three times a turn that's already carrying tens of thousands of tokens of prior context. Model choice multiplies whatever that baseline turns out to be.

That cost structure is pushing people to split the loop apart. One example: a "Refine-Plan-Act" pattern divides a coding task into three separate phases — refine the requirements into a document, plan the implementation into a second document, then act — specifically so a misunderstood requirement gets caught in the requirements document instead of discovered mid-implementation, and so a bad implementation can be discarded while the validated plan underneath it survives. The phases are kept as separate, reviewable artifacts rather than folded into one continuous agent context.

Steve Yegge's Gas Town takes a different cut at the same problem. Rather than splitting a single agent's loop into phases, Gas Town splits the work across role-specialized agents — a coordinating "Mayor," worker "Polecats," monitoring "Witnesses" — with state persisted externally in a Git-backed issue tracker called Beads rather than carried in any one agent's context window. Five months after launch, Yegge noted that newer models hadn't reduced the need for this structure the way he'd originally expected; the orchestration stayed load-bearing as the underlying models got stronger, rather than becoming unnecessary scaffolding.

Both of these are responses to the same underlying problem with the plain loop: judgment and execution, or planning and implementation, end up bundled into one actor with no separation of concerns and no external record of what it decided or why. Ratchet is a third cut at that same problem, descended directly from a personal adaptation of Gas Town's Mayor/Polecat pattern, built to find out whether a more granular version of the role split holds up against a real fleet of local models rather than a single frontier model doing everything.

## What Ratchet is

The Gas Town adaptation that Ratchet grew out of used Claude Desktop as the Mayor — the orchestrating layer — and open models served locally through Ollama as Polecats, the workers executing individual tasks. That two-role split was a reasonable starting point, but it ran into a structural problem during early testing: a Mayor that's free to alter a decomposition during planning, with no mechanism positioned to catch when it does, can introduce errors that no Polecat is in a position to notice. Two such Mayor-introduced errors — a fixed-quota encoding scheme replacing an intentionally opportunistic one, and an indexing scheme replacing a flat-string one — invalidated an early experiment entirely, not because any worker model failed, but because the orchestrating layer made an unjustified change upstream of where any check existed.

That failure is the direct origin of Ratchet's architecture. A role like "Mayor" or "Polecat" is a noun: it names an actor, not the action that actor is performing at a given moment, and a single noun can quietly cover several actions with different correctness criteria. This isn't unique to one adaptation — it's how Gas Town's own published role list is structured. The Mayor's documented job spans analyzing a request and breaking it into tasks, creating the unit that tracks the work, spawning agents, distributing work to them, monitoring their progress, and summarizing completion: six distinct actions under one name. The Witness's documented job spans detecting stuck agents, triggering recovery, managing session cleanup, and tracking completion: four more actions under a different single name. The Deacon checks agent health, dispatches maintenance workers, and escalates issues it can't resolve itself. None of these role definitions distinguish which of their bundled actions is succeeding or failing at any given moment — the role reports as a whole, or not at all.

When a role bundles several actions, there's no natural seam at which to ask whether the same actor should be doing all of them, or whether one of them needs an independent check. That's exactly what went wrong in the Mayor-introduced decomposition errors above: "decide how to break the work down" and "decide whether that breakdown is correct" were the same actor's job, with nothing positioned to ask whether they should be.

Ratchet replaces role nouns with verb identities. There are eight: `DECOMPOSE_SPEC`, `AUDIT_DECOMPOSITION`, `RECONCILE_DECOMPOSITION`, `EXECUTE_BEAD`, `MONITOR_EXECUTION`, `ANALYZE_EXECUTION`, `COMPRESS_ANALYSIS`, and `ADJUDICATE_NEXT_EXECUTION`. Each verb is defined by what would make its output correct, not by which actor performs it or what shape its input and output happen to take. Two verbs can be handled by the same model and still be different verbs; the same model handling both is never evidence that they should be merged into one.

The pipeline runs in two phases. A decomposition phase turns a design document into discrete units of work and checks that breakdown before anything gets built. An execution phase carries each unit through implementation, monitoring, analysis, and a decision about what happens next. Both phases share the same shape: a bounded loop for resolving disagreement, and a single escalation path to a person when that loop runs out.

This distinction has a concrete consequence: `DECOMPOSE_SPEC` (breaking a design document into discrete units of work, called Beads) and `AUDIT_DECOMPOSITION` (checking whether that breakdown is sound) are required to run on different models. The model that wrote a decomposition reviewing its own work is a weaker check than a second model reviewing it — self-review has repeatedly missed findings that a different model, looking at the same material, caught. If `AUDIT_DECOMPOSITION` finds a problem, a third verb, `RECONCILE_DECOMPOSITION`, resolves the disagreement. That resolution is bounded rather than open-ended: capped at two rounds, with the full history of the disagreement preserved rather than summarized away, and convergence detected mechanically by the queue managing the process rather than by either model deciding for itself that it's done. If the two rounds run out without resolution, the whole decomposition escalates to a person rather than continuing to iterate.

The same logic governs execution. `EXECUTE_BEAD` performs the work; `MONITOR_EXECUTION` watches it run, as a concurrent process rather than a step waiting for execution to finish; `ANALYZE_EXECUTION` interprets what happened after the fact; `COMPRESS_ANALYSIS` reduces a growing history of analysis into something a later decision can act on without re-reading everything; `ADJUDICATE_NEXT_EXECUTION` decides what happens next. Each of these is a distinct correctness criterion, not a distinct actor — multiple verbs frequently run on the same model, and that overlap is never treated as a reason to fold them together.

A finite state machine sits underneath all of this as the actual source of truth. A bead's status isn't a story a model tells about its own progress; it's a state transition, recorded in storage, in the same transaction as the signal that work is complete. Each project is its own first-class entity in that storage, holding its own verb-to-model assignments and its own configured defaults, so a project's choices about which model handles which verb are data the system can read, not decisions baked into a deployed binary.

*(Role descriptions above are from Gas Town's published documentation: [github.com/gastownhall/gastown](https://github.com/gastownhall/gastown).)*

None of this is presented here as proven. It's a description of what was built and the design principle it was built around — gating verb identity on correctness criteria rather than on which actor performs the work, or what shape the work's inputs and outputs happen to take. Whether that principle actually held up — where it predicted the right thing, where the experiments overturned an assumption, and where the architecture had a bug in its own reasoning that a section-by-section adversarial review caught — is a separate question, and the subject of the next piece.
